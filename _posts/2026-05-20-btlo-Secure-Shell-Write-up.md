## Walkthrough

### Q1: Is it an internal or external attack, what is the attacker IP?

Let's first find out where the attack is coming from. 

```jsx
host="BTLO" index="btlo_secure_shell_hard"
| rex "from (?<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count by src_ip
| sort -count
```

We'll extract all IP addresses from the SSH logs and count how many requests came from each one.

![image.png](/assets/img/posts/btlo-secureshell/image1.png)

From IP that we found I used IPabuse to verify if the ip is internal or external.

![image.png](/assets/img/posts/btlo-secureshell/image2.png)

Now, we have verify the IP is external based on the reporter.

<aside>
💡

Answer: **external:192.168.1.17**

</aside>

### Q2: How many valid accounts did the attacker find, and what are the usernames?

Now that we've identified the attacker, let's see if they managed to compromise any accounts. 

We only need to look for **Accepted password** events, which indicate successful logins. using below query:

```jsx
host="BTLO" index="btlo_secure_shell_hard" "Accepted password"
| rex "Accepted password for (?<username>\S+)"
| stats values(username)
```

![image.png](/assets/img/posts/btlo-secureshell/image3.png)

it’s turn put was sophiaaa.

<aside>
💡

Answer: **sophia**

</aside>

### Q3: How many times did the attacker login to these accounts?

Now let's find out how many times the attacker successfully logged into the compromised account.

```jsx
| rex "Accepted password for (?<username>\S+)"
| stats count by username
```

We'll count every successful login event for the username we found earlier.

### Q4: When was the first request from the attacker recorded?

Every attack starts somewhere. Let's identify the earliest request made by the attacker to determine when the activity first began.

```jsx
host="BTLO" index="btlo_secure_shell_hard" | rex "from (?<src_ip>\d{1,3}(?:\.\d{1,3}){3})"
| stats earliest(_time) as first_request by src_ip
| eval first_request=strftime(first_request,"%Y-%m-%d %H:%M:%S")
| sort first_request
```

![image.png](/assets/img/posts/btlo-secureshell/image5.png)

From the time I make add query `23:52:25`  to specify in detail the timing.

```jsx
host="BTLO" index="btlo_secure_shell_hard" | rex "from (?<src_ip>\d{1,3}(?:\.\d{1,3}){3})" | search src_ip="192.168.1.17" "23:52:25"
```

![image.png](/assets/img/posts/btlo-secureshell/image6.png)

<aside>
💡

Answer: **2021-04-29 23:52:25.989**

</aside>

### Q5: What is the log level for the log file?

```jsx
host="BTLO" index="btlo_secure_shell_hard"
| rex "(?<log_level>debug\d+|info|notice|warning|error|fatal)"
| stats count by log_level
```

![image.png](/assets/img/posts/btlo-secureshell/image7.png)

Before answering this question, let's quickly understand what **SSH LogLevel** means.

In SSH, the LogLevel option allows you to control the level of logging information generated by the SSH client and server.

There are several log levels you can use to adjust the verbosity of SSH logging. Here are the most commonly used log levels:

| **Log Level** | **Description** |
| --- | --- |
| **QUIET** | Suppresses all log messages except fatal errors. |
| **FATAL** | Logs only fatal errors that prevent the SSH session from being established. |
| **ERROR** | Logs error messages that may affect the connection. |
| **INFO** | Logs general information such as connection status and key exchange. *(Default level)* |
| **VERBOSE** | Provides more detailed information than **INFO**. |
| **DEBUG** | Generates detailed debugging messages for troubleshooting connection and authentication issues. |
| **DEBUG1** | Provides detailed debugging output. |
| **DEBUG2** | Provides more detailed debugging information than **DEBUG1**. |
| **DEBUG3** | **Provides the most detailed debugging output.** |

<aside>
💡

Answer: **debug3**

</aside>

### Q6: Where is the log file located in Windows?

If you're wondering where Windows stores SSH logs by default, they're located under the **ProgramData** directory.

<aside>
💡

Answer: **C:\ProgramData\ssh\logs\sshd.log**

</aside>
