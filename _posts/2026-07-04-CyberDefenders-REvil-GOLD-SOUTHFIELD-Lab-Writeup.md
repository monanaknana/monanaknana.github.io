## **Scenario**

You are a Threat Hunter working for a cybersecurity consulting firm. One of your clients has been recently affected by a ransomware attack that caused the encryption of multiple of their employees' machines. The affected users have reported encountering a ransom note on their desktop and a changed desktop background. You are tasked with using Splunk SIEM containing Sysmon event logs of one of the encrypted machines to extract as much information as possible.

## Walkthrough

### Q1: To begin your investigation, can you identify the filename of the note that the ransomware left behind?

I first identified the available Sysmon event types to determine which event was most relevant for this investigation as below. Since ransomware typically creates a ransom note after encrypting files, Event ID 11 (File Create) was the best starting point.

```jsx
index=revil
| stats count by event.code
| sort -count
```

![image.png](/assets/img/posts/cb-REvil/image1.png)

Next, I reviewed the earliest file creation events by displaying the target filenames in chronological order.

```jsx
index=revil event.code=11
| table winlog.event_data.UtcTime host.name winlog.computer_name winlog.event_data.Image winlog.event_data.ProcessId winlog.event_data.User winlog.event_data.TargetFilename winlog.event_data.RuleName
| reverse
```

After examining the results, I noticed an unusual text file that matched the naming pattern of a ransomware note.

![image.png](/assets/img/posts/cb-REvil/image2.png)

<aside>
💡

Answer: 5uizv5660t-readme.txt

</aside>

---

### Q2: After identifying the ransom note, the next step is to pinpoint the source. What's the process ID of the ransomware that's likely involved?

Once the ransom note was identified, I examined the same File Create event to determine which process created the file. The event contains the ProcessId responsible for the file creation. 

![image.png](/assets/img/posts/cb-REvil/image3.png)

Since the ransom note was generated by the ransomware, this Process ID is most likely associated with the malicious executable.

<aside>
💡

Answer: 5348

</aside>

---

### Q3: Having determined the ransomware's process ID, the next logical step is to locate its origin. Where can we find the ransomware's executable file?

Time is so much important when we threat hunting.

Since the ransom note was created by Process ID 5348, I used its timestamp as a reference for the next investigation. 
I then moved on to Sysmon Event ID 1 (Process Create), which records details about every process execution, including the executable path, parent process, and command line. By reviewing the events around the same timestamp, I was able to trace the process back to the ransomware executable.

![image.png](/assets/img/posts/cb-REvil/image4.png)

To identify the executable, I used the following query:

```jsx
index=revil event.code=1
| table winlog.event_data.UtcTime host.name winlog.computer_name winlog.event_data.Image winlog.event_data.User winlog.event_data.ParentImage
| reverse
```

![image.png](/assets/img/posts/cb-REvil/image5.png)

<aside>
💡

Answer: C:\Users\Administrator\Downloads\facebook assistant.exe

</aside>

---

### Q4: Now that you've pinpointed the ransomware's executable location, let's dig deeper. It's a common tactic for ransomware to disrupt system recovery methods. Can you identify the command that was used for this purpose?

After locating the ransomware process, I expanded the related event details to inspect its command-line arguments.

![image.png](/assets/img/posts/cb-REvil/image6.png)

 I found a Base64-encoded PowerShell command, which is commonly used to hide malicious activity. After decoding it using **CyberChef** with **UTF-16LE (Unicode)** encoding, the original PowerShell command was revealed.

![image.png](/assets/img/posts/cb-REvil/image7.png)

<aside>
💡

Answer: Get-WmiObject Win32_Shadowcopy | ForEach-Object {$_.Delete();}

</aside>

---

### Q5: As we trace the ransomware's steps, a deeper verification is needed. Can you provide the sha256 hash of the ransomware's executable to cross-check with known malicious signatures?

To verify the malware, I examined the Process Create event for the ransomware executable. 

![image.png](/assets/img/posts/cb-REvil/image8.png)

Sysmon records file hashes when configured, allowing analysts to identify malware uniquely. The SHA256 hash can then be used to search threat intelligence platforms such as VirusTotal or MalwareBazaar for additional information. in this cases I using Virustotal to solve this.

![image.png](/assets/img/posts/cb-REvil/image9.png)

<aside>
💡

Answer: B8D7FB4488C0556385498271AB9FFFDF0EB38BB2A330265D9852E3A6288092AA

</aside>

---

### Q6: One crucial piece remains: identifying the attacker's communication channel. Can you leverage threat intelligence and known Indicators of Compromise (IoCs) to pinpoint the ransomware author's onion domain?

Using the SHA256 hash obtained in the previous step as an Indicator of Compromise (IoC), I searched several threat intelligence platforms, including **VirusTotal**, **ANY.RUN**, and **MalwareBazaar**. These platforms provided references to malware reports and related IoCs.

I then reviewed a **Sophos** threat report, which contained several onion domains associated with the ransomware family.

[https://www.sophos.com/en-us/research/lv-ransomware](https://www.sophos.com/en-us/research/lv-ransomware) - which i found I at Virus total but it not works.

![image.png](/assets/img/posts/cb-REvil/image10.png)

Since the listed domains did not match the challenge, I continued my investigation using a **Triage** malware analysis report. From the ransom note displayed in the report, I identified the attacker's Tor payment portal -  [https://tria.ge/260212-qazsyabv9d](https://tria.ge/260212-qazsyabv9d)

![image.png](/assets/img/posts/cb-REvil/image11.png)

fter scrolling through the report, I located the onion domain associated with the ransomware.

![image.png](/assets/img/posts/cb-REvil/image12.png)

<aside>
💡

Answer: aplebzu47wgazapdqks6vrcv6zcnjppkbxbr6wketf56nf6aq2nmyoyd.onion

</aside>
